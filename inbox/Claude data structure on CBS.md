A persistent data dictionary at the project level. Not just variable names and types, but the quirks — coverage gaps, known anomalies, how certain variables are constructed, what merges are problematic. The kind of thing you'd tell a new RA on day one. Claude Code can maintain this as a living document that gets referenced at the start of each session. That way you're not re-explaining that the incarceration variable has 295 false zeros every time something goes wrong.
A separate "known issues and ruled out explanations" log. When you hit a problem and establish what it's not, that goes in the log explicitly. Then when Claude starts down a familiar wrong path you can just reference it — "we already ruled this out, see the log" — rather than relitigating it.
And when you paste error output, adding a short header with the relevant data context — sample size, which merge stage you're at, which variables are involved — gives Claude enough to reason from rather than guessing at the data structure.
